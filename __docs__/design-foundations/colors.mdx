## Semantic tokens

Semantic tokens are the color variables designers and engineers use to apply color across Khan Academy experiences. Rather than referencing raw hex values, tokens are named by intent — what a color does, not what it looks like. This makes it easier to apply color consistently, maintain accessibility, and update the system at scale.

<img src={semanticTokensImg} alt="Semantic tokens" />

Tokens are organized by three properties:

-   **Layer** — what element you're coloring (background, foreground, border, or shadow)
-   **Context** — the meaning or purpose (instructive, success, critical, neutral, etc.)
-   **Intensity** — the visual weight (subtle, default, or strong)

When building in Figma, always apply color using semantic tokens rather than raw values or primitives.

[Deep dive: Color tokens on Confluence](https://khanacademy.atlassian.net/wiki/spaces/WB/pages/3683221530/Color#Themes-and-Palettes)

## Usage guidelines

### Tokens for everything

Every color applied in a design — backgrounds, text, icons, borders, shadows — should use a semantic token. If you're reaching for a color swatch or hex value directly, stop and find the token instead.

### Intent over appearance

Tokens are named for what they do, not what they look like. Choose based on the purpose of the element: guiding a primary action is Instructive, confirming a completed action is Success, signaling an error or destructive action is Critical. If the intent fits, the token is right.

### No primitives or raw hex

Primitive tokens and hex values are the raw ingredients that feed the token system — they're not meant to be applied directly in designs. Using them bypasses the system and breaks consistency across themes and components.

### Matching intent is required

A token that looks correct but carries the wrong meaning is still the wrong token. If you're using a Critical token because the red looks good, that's a misuse — users and the system will interpret it as an error state.

## Accessible pairings

All color combinations must meet minimum contrast ratios. The applicable contrast ratio depends on the content type. Purely decorative elements with no informational purpose are exempt.

<img src={accessiblePairingsImg} alt="Accessible pairings" />

### 3:1 contrast ratio

-   Large Text (19–23px + bold weight | ≥ 24px)
-   Distinguishing borders or fills of controls
-   Critical graphics

### 4.5:1 contrast ratio

-   Normal Text (≤ 18px)

### Recommended combinations

The token system is built with accessible pairings in mind.

**Background**

-   Default and Strong Background tokens (minus Warning) pass at 3:1+ on the Default and Subtle Base Background tokens
-   Strong Background tokens pass at 3:1+ on all Subtle Background tokens

**Borders**

-   Strong Border tokens pass at 3:1+ on all Subtle Background tokens and the Default Base Background token
-   Default and Subtle Border tokens (minus Warning) pass at 3:1+ on all Subtle Background tokens and the Default Base Background token

**Foreground**

-   Strong Foreground tokens pass at 4.5:1+ on all Subtle Background and Base Background tokens
-   Default Foreground tokens pass at 4.5:1 on Default and Subtle Base Background tokens and most Subtle Background tokens (with a few 3:1 only passing exceptions)

Following the usage guidelines above will help keep you within compliant combinations.

### Working with color

We ensure that users can distinguish between elements and understand any instructions, states, or other design cues by pairing color changes with non-color cues. These can include:

-   Adding / applying a new icon, underline, border, shadow, or glow
-   Increasing font weight, underline, border, or stroke thickness
-   Increasing the spacing of / between elements, including letter spacing
-   Including patterns or shapes with data visualizations
